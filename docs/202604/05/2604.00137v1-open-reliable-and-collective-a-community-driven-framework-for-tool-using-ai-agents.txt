# Open, Reliable, and Collective: A Community-Driven Framework for Tool-Using AI Agents

Abstract 

Tool-integrated LLMs can retrieve, compute, and take real-world actions via external tools, but reliability remains a key bottleneck. We argue that failures stem from both tool-use accuracy (how well an agent invokes a tool) and intrinsic tool accuracy (the tool’s own cor-rectness), while most prior work emphasizes the former. We introduce OPEN TOOLS 1, a community-driven toolbox that standardizes tool schemas, provides lightweight plug-and-play wrappers, and evaluates tools with auto-mated test suites and continuous monitoring. We also release a public web demo 2 where users can run predefined agents and tools and contribute test cases, enabling reliability re-ports to evolve as tools change. O PEN TOOLS 

includes the core framework, an initial tool set, evaluation pipelines, and a contribution protocol. Experiments and evaluations show improved end-to-end reproducibility and task performance; community-contributed, higher-quality task-specific tools deliver 6%–22% rel-ative gains over an existing toolbox across mul-tiple agent architectures on downstream tasks and benchmarks, highlighting the importance of intrinsic tool accuracy. The source code is released under the Apache 2.0 license. We also provide a system demonstration video at the following link: https://www.youtube.com/ watch?v=MXVyDvXzh_o .

1 Introduction 

Large language models (LLMs) have rapidly evolved from purely generative systems into general-purpose “agents” that can plan, reason, and interact with users over extended contexts (Grattafiori et al., 2024). Recent agentic frame-works show that LLMs can decompose high-level goals into multi-step workflows through organized

Despite progress in agentic LLMs, token-only generation remains brittle when tasks require fresh knowledge or grounded interaction with exter-nal systems (Paranjape et al., 2023; Qin et al., 2023), often leading to errors and hallucinations in domain-specific settings (Huang et al., 2024; Kokane et al.; Agrawal et al., 2024). Tool-augmented language models (TALMs) address this by letting LLMs delegate problems to external tools (e.g., retrieval (Gao et al., 2023; Jin et al., 2025)), calculators (Schick et al., 2023), code interpreters (Wang et al., 2024a), and domain APIs in chem-istry/physics/medicine (Wu et al., 2025; Arlt et al., 2025; Jang et al., 2025). This delegation improves factuality and extends agents to real-world tasks. We argue that TALM reliability has two failure modes: (i) tool-use accuracy (choosing and invok-ing tools correctly) and (ii) intrinsic tool accuracy 

(tool correctness, stability, and robustness to drift). However, today’s tool ecosystem is fragmented: most work emphasizes tool learning (Qin et al., 2024; Patil et al., 2024) while largely assuming tools and documentation are reliable. In practice, tool-side failures are common such as missing cov-erage, API changes, nondeterminism, silent errors, and version drift, and tool access can introduce se-curity risks (Winston and Just, 2025; Milev et al., 2025). This undermines reproducibility because systems may break across tool versions or environ-ments even when agent policies are unchanged, and continuous monitoring and testing is rarely avail-able. Moreover, there is no widely adopted open framework for community contribution of tools and stress tests, or for continuously updating reliability signals as tools evolve. 1

To address this gap, we introduce O PEN TOOLS ,an open-source, community-driven toolbox for tool-integrated LLMs. OPEN TOOLS (i) standardizes tool schemas and lightweight wrappers for plug-and-play use across agent frameworks, (ii) provides an intrinsic tool evaluation suite that tracks correct-ness, robustness, and regressions via community-contributed tests, and (iii) releases a public demo interface and contribution protocol to support iter-ative updates of tools and reliability reports. The demo interface lowers the barrier to community feedback when local setup is impractical (e.g., de-pendencies, API keys), accelerating iteration on tool reliability and agent behavior. Across diverse domains, our experiments show that higher-quality, task-specific tool coverage in O PEN TOOLS yields 

6%–22% relative gains over an existing toolbox across multiple agent architectures. We hope O PEN TOOLS enables open, reliable, continuously maintained tool ecosystems for repro-ducible, trustworthy research and deployment. 

2 OPEN TOOLS Framework 

OPEN TOOLS is an open-source, community-driven framework for building, evaluating, and deploying tools for tool-integrated language models. It op-erationalizes the reliability distinction introduced above by providing two workflows: (1) a Tool Accuracy/Maintenance Loop for continuous val-idation and regression testing, and (2) an Agentic Workflow for integrating tool collections into LLM agents to solve user tasks as depicted in Figure 1. OPEN TOOLS is guided by four design goals: 

Standardization Tools should be defined with a consistent schema so they can be plugged into diverse agent frameworks with minimal adaptation. 

Continuous, community-driven reliability eval-uation Tool quality should be measurable and monitored over time. O PEN TOOLS makes intrinsic tool accuracy first-class by supporting tests, robust-ness checks, and repeatable evaluation reports. 

Separation of concerns between tools and agents 

Tool maintenance and tool orchestration should be decoupled: tools can be improved and re-tested independently of agent policies, and agents can be swapped without rewriting tools. 

Transparency and debuggability. Tool evalua-tion and agent runs emit structured logs to support debugging, reproducibility, and failure analysis. 

2.1 Tool Accuracy / Maintenance Loop 

In tool-integrated LLM systems, tools are often assumed to be reliable. However, tool behavior can change over time due to API updates, backend changes, nondeterminism, rate limits. As a result, an agent may select the correct tool and arguments but still obtain incorrect or unstable outputs. Work-flow 1 therefore treats intrinsic tool accuracy as a primary objective and introduces an evaluation-and-maintenance loop that continuously measures and updates tool reliability signals. 

Problem Definition: Given a tool τ and a suite of test cases T , the goal is to estimate and maintain an up-to-date reliability profile R(τ ) that reflects the tool’s correctness and robustness under realistic conditions. O PEN TOOLS treats this as an ongoing, community-driven regression testing task, where test suites evolve over time and reliability estimates are recomputed as tools or environments change. 

Workflow Overview. The tool-accuracy work-flow forms a closed loop with two phases: (i) relia-bility evaluation and (ii) community maintenance .

2.1.1 Tool Reliability Evaluation Tool Standardization. To enable consistent eval-uation across heterogeneous tools (e.g., calculators, retrievers, API wrappers), O PEN TOOLS uses a uni-fied interface consisting of (1) a natural-language tool description, (2) a typed argument schema with 

JSON-serialized parameter inputs and (3) a struc-tured output contract. Different from OctoTools 

(Lu et al., 2025), which relies on toolbox-specific metadata and formats, O PEN TOOLS adopts this common JSON-parameter convention to ensure consistent invocation across tools and agent frame-works. Tools may also expose an optional intrinsic-accuracy summary from prior evaluations (e.g., current-suite accuracy), enabling comparable relia-bility signals before usage. 

Test Suites and Verification. OPEN TOOLS 

maintains a repository of test cases specifying ex-pected tool behavior by pairing each tool with an input and an expected output (or verifiable prop-erty), optionally with tool-specific checks (e.g., pat-tern matching). A verification module then scores tool outputs using task-appropriate criteria (exact match, tolerance, property validation, or semantic equivalence). 2.1.2 Community-Driven Maintenance 

Tool reliability is non-stationary: APIs evolve, de-pendencies drift, and new edge cases arise in de-ployment. OPEN TOOLS treats test suites as liv-ing artifacts maintained by the community. Users submit test cases that reproduce failures or ex-pand coverage; accepted cases become part of the shared benchmark and run in subsequent evaluation rounds. Reliability reports are thus continuously updated, enabling tracking and monitoring. 

Tool Contributions. Beyond tests, users can con-tribute new tools through the shared repository. All tools follow the same standardized schema, en-abling immediate evaluation against curated and community suites and providing an initial reliabil-ity snapshot upon release. 

2.2 Agentic Workflow 

Whereas the Maintenance Loop focuses on intrinsic tool reliability, the agentic workflow uses O PEN -TOOLS as a toolbox for tool-integrated LLM agents. It packages tool collections for general use across agent frameworks and provides end-to-end execu-tion with transparent, debuggable traces. 

Problem Definition. Given a user query q, a tool-box S, and an agent policy π (e.g., prompting-based planning/routing or custom control flow), the agentic workflow produces an answer y by execut-ing tool calls and intermediate reasoning steps. It also outputs an execution trace L that records tool usage and outcomes for reproducibility and failure diagnosis: (q, S, π ) → (y, L).

Workflow Overview. The agentic workflow has two phases: (i) Agent Setup and Tool Access and 

(ii) Interactive Task Execution .

2.2.1 Agent Setup and Tool Access 

The setup phase specifies which tools the agent can use and how they are exposed. 

Toolbox Interface. OPEN TOOLS wraps hetero-geneous tools behind a unified interface (descrip-tions and schemas), enabling invocation without tool-specific glue code and integration into external agent frameworks via lightweight adapters. 

Tool Subsetting and Prioritization. Users can restrict the agent to task-relevant tools. When avail-able and enabled, reliability signals from Work-3flow 1 can guide selection or routing (e.g., prefer higher-accuracy tools or avoid regressed ones). 

Agent Policy Selection. OPEN TOOLS includes several standard agent policies (e.g., ReAct (Yao et al., 2023b), OctoTools (Lu et al., 2025), and MultiAgent) and supports user-defined agents via custom controllers and workflows. The policy mod-ule determines action selection, and policies are treated as plug-and-play components, decoupling 

tool access from decision logic .

2.2.2 Interactive Task Execution 

During execution, the agent iteratively updates the task state, invokes tools, and produces the final an-swer. An execution manager mediates between the policy and toolbox: it validates arguments against schemas, executes tool calls, returns results as ob-servations, and records failures (e.g., exceptions, rate limits, timeouts). O PEN TOOLS logs a struc-tured trace of tool invocations, inputs/outputs, and error states, enabling reproducibility and attribu-tion of failures to either policy errors (tool-use) or tool-side issues (intrinsic reliability), and linking behavior to Workflow 1 reliability signals. The system outputs the answer and trace for interactive inspection and offline analysis. 

2.3 Web Demonstration Interface 

Beyond the open-source repository, O PEN TOOLS 

provides a public web demonstration interface 3

designed to lower the barrier for community par-ticipation. The demo exposes the two workflows described above: (1) the tool accuracy / mainte-nance loop and (2) the agentic workflow, through an interactive UI that requires no local setup and enables rapid testing, feedback, and contribution. We further provide additional details of the Web Demonstration Interface in Appendix A.1 

Tool Accuracy Interaction. For Workflow 1, the web interface lets users (i) browse tools and inspect tool cards (description, argument schema, usage notes), (ii) run tools on custom inputs, and (iii) view each tool’s current intrinsic accuracy when available (from the latest evaluation report). Crucially, users can turn observed failures into tests : they contribute new test cases by specifying inputs and expected outputs (or verifiable proper-ties) and optionally annotating mismatches. Ac-cepted cases are added to the shared suite and exe-

Agentic Workflow Interaction. For Workflow 2, the demo provides an end-to-end sandbox where users can input a task query, optionally restrict the tool subset, and run a predefined agent configura-tion (e.g., ReAct (Yao et al., 2023b), OctoTools (Lu et al., 2025)). By integrating contribution (tests and feedback), evaluation (accuracy reporting), and ex-ecution (agent runs) in a single interactive envi-ronment, the demo operationalizes our community-driven goal and streamlines the loop from observed failures to cover test cases and tool improvements. 

3 Experiments 

We evaluate O PEN TOOLS under the agentic work-flow to quantify how a community-maintained tool-box impacts end-to-end task performance across multiple agent frameworks. Concretely, we com-pare agent performance when equipped with the O PEN TOOLS toolbox (including community-contributed tools and tests) versus a strong, cu-rated toolbox baseline (OctoTools toolbox). Our evaluation targets breadth (many benchmarks and domains) and generality (multiple agent policies), while keeping the underlying base LLM fixed.

3.1.2 Baselines Toolboxes. We compare two toolboxes: (i) OctoTools-T A toolbox from the OctoTools testbed design, including a generalist solution generator and a set of domain-specific tools (e.g., image captioning, web search, python code generation, specialized classifiers) and (ii) O PEN TOOLS -T 

Our community-driven toolbox consisting of (i) a core set of general-purpose tools (e.g., search, calculator/code execution), (ii) collected domain tools (e.g., vision/perception, chemistry helpers), and (iii) community-contributed tools (e.g., maze solvers). Tools are exposed through a standard-ized interface and are continuously evaluated using Workflow 1. The more detailed information of the ToolBoxes are represented in Appendix A.2 

Agent Frameworks. We evaluate our systems under multiple agent frameworks: (i) Prompting: 

the LLM answers directly without tool use. We ex-periment with both zero-shot and chain of thought prompting (CoT), which are step-by-step reasoning without external tools). In preliminary experiments, zero-shot performs better overall for both gpt-4o-mini and gpt-5-mini; therefore, we report zero-shot as Prompting in the main results and provide the full zero-shot/CoT comparison in Appendix A.2. 

(ii) ReAct Agent (Yao et al., 2023b): an itera-tive reason-and-act agent that interleaves natural-language reasoning with tool calls. (iii) OctoTools Agent (Lu et al., 2025): the planner–executor style agent as defined in the OctoTools framework. (iv) MultiAgent Framework: Inspired by LemonA-gent (Jiang et al., 2026), we use a hierarchical pipeline that decomposes a query and routes sub-problems to specialized sub-agents. Unlike Lemon-Agent, we (i) perform explicit reasoning-based decomposition before routing and (ii) use a Ver-ifier /guardrail to store only validated intermediate results in shared memory. A Planner decomposes the query, a Generator issues tool calls, tools are executed and verified, validated results are written to memory, and a Composer generates the final answer with a brief memory-based summary. For tool-using agents, we run each framework with (i) the OctoTools toolbox (OctoTools-T) and (ii) our O PEN TOOLS toolbox (OpenTools-T), en-abling a controlled comparison of toolsets under the same decision logic. Following OctoTools (Lu et al., 2025), we perform greedy tool selection for OpenTools-T to choose task-appropriate tool sub-sets. For OctoTools-T, we use the tool sets reported in the OctoTools paper that achieve the best perfor-mance for each task.

5 Related Work 

Tool use and agentic LLMs. Prior work on tool use and agentic LLMs focuses on how models plan, and invoke external tools through training or prompting, including Toolformer-style learn-ing (Schick et al., 2023), ReAct reasoning-and-acting (Yao et al., 2023b), and agent controllers that add reflection/search and multi-step tool exe-cution (Patil et al., 2024; Dang et al., 2025). These lines of work primarily improve tool-use accuracy, while often assuming the underlying tools are cor-rect and stable. In parallel, tool libraries and evaluations (e.g., LangChain 4, AutoGen 5) broaden tool access and provide domain-specific suites (e.g., chemistry (Wu et al., 2025), medical (Li et al., 2024a), vision (Hu et al., 2024)). Existing benchmarks assess end-to-end task success and tool-calling competence (Huang et al., 2023; Guo et al., 2024), but fre-quently conflate model mistakes with tool-side fail-ures and seldom measure tool drift or regressions over time. O PEN TOOLS addresses this infrastruc-ture gap by standardizing JSON-schema tool inter-faces, providing community-maintained wrappers, and enabling intrinsic tool evaluation with relia-bility signals and regression tracking—supporting plug-and-play use across agent frameworks while separating tool-use from tool reliability. 

6 Conclusion and Future Work 

We introduced O PEN TOOLS , a community-driven toolbox and evaluation framework for tool-integrated LLM agents that explicitly separates 

tool-use accuracy from intrinsic tool accuracy .OPEN TOOLS standardizes tool interfaces and wrap-pers for plug-and-play use across agent frame-works, and provides an evaluation suite that tracks tool correctness, robustness, and regressions over time. Our experiments show that improving tool quality and task coverage yields consistent down-stream gains across multiple agent policies, with the largest benefits on tool-intensive end-goal tasks. 

Future work. We plan to expand O PEN TOOLS 

in three directions: (1) broaden tool coverage by curating additional tools and supporting commu-nity contributions, with emphasis on specialized domains (e.g., science, medicine, engineering); (2) strengthen intrinsic reliability benchmarking via more stress tests and real-world failure cases, im-proved regression tracking, and continuous evalu-ation as APIs evolve; and (3) keep pace with ad-vancing agent frameworks by adding pre-defined agent policies and reference implementations so the same tool ecosystem can be evaluated and de-ployed under emerging architectures.

Tool provenance varies: some tools are newly implemented to support more complex capabilities and controlled evaluation through a unified call-ing interface. Others are adapted from existing open-source ecosystems (e.g., OctoTools (Lu et al., 2025), AWorld (Yu et al., 2025)) and refactored to enforce consistent schemas and use across agent policies. Overall, O PEN TOOLS aims to provide a community-extendable toolset whose end-to-end gains depend on both orchestration quality and the intrinsic correctness and stability of the tools. 11
