# Evaluating LLM-generated code for domain-specific languages: molecular dynamics with LAMMPS 

# Ethan W. Holbrook a, ∗, Juan C. Verduzco a, ∗, and Alejandro Strachan aa School of Materials Engineering and Birck Nanotechnology Center Purdue University, West Lafayette, Indiana, 47907 USA * Authors contributed equally to this work. 

Abstract 

Large language models (LLMs) are changing the way researchers interact with code and data in scientific computing. While their ability to generate general-purpose code is well established, their effectiveness in producing scientifically valid code/input scripting for domain-specific languages (DSLs) remains largely unexplored. We propose an evaluation procedure that enables domain experts (who may not be experts in the DSL) to assess the validity of LLM-generated input files for LAMMPS, a widely used molecular dynamics (MD) code, and to use those assessments to evaluate the performance of state-of-the-art LLMs and identify common issues. Key to the evaluation procedure are a normalization step to generate canonical files and an extensible parser for syntax analysis. The following steps isolate common errors without incurring costly tests (in time and computational resources). Once a working input file is generated, LLMs can accelerate verification tests. Our findings highlight limitations of LLMs in generating scientific DSLs and a practical path forward for their integration into domain-specific computational ecosystems by domain experts. 

Keywords - large language models; artificial intelligence; computational simulations; molecular dynamics 

# Introduction 

Physics-based simulations are a staple in modern research and the development and availability of well-documented, verified, and efficient code, often tuned to take advantage of modern hardware, has firmly established computational science as a discipline. Examples include Quantum Espresso 1

and VASP 2 for density functional theory calculations, LAMMPS 3 and NAMD 4 for molecular dy-namics (MD), and MOOSE 5 for finite elements. Running these tools requires knowledge of each software’s ad hoc domain-specific language (DSL), the specialized interface through which numer-ical algorithms are translated into simplified commands describing physical models. 6 These DSLs are often syntactically complex. For example, even small deviations in command order or structure can invalidate a simulation. Errors often emerge when critical details are left implicit by relying on default values or through incomplete descriptions of scripts, an issue compounded by the scarce sharing of these input scripts. The associated semantic complexity, in contrast, reflects the need 1

for a deep understanding of the underlying physics to construct meaningful and physically consis-tent simulation instructions. Together, these challenges hinder reproducibility, accessibility, and broader adoption. The pace of scientific discovery is slowed as a consequence. Bridging human intent with accurate and reproducible simulation workflows requires new strategies that reconcile natural language with the strict syntax of DSLs. LAMMPS, the Large-scale Atomic/Molecular Massively Parallel Simulator, illustrates both the promise and the limitations of this paradigm. With its wide applicability and excellent performance in multiple platforms, LAMMPS has unified a significant portion of the computational materials science community and lowered the barrier to running complex MD workflows. 3 Its input language allows researchers to harness powerful codes while shielding them from the complexity of the un-derlying algorithms, making simulations more accessible. Despite its impact, LAMMPS exemplifies the tension between flexibility and fragility in scientific DSLs: its ad hoc and semi-structured syntax mixes command dependencies and defaults in ways that make input files error-prone and difficult to validate, particularly in the absence of supporting infrastructure for error handling such as parsers or linters. While researchers are generally able to articulate the simulations they wish to run, they may lack the expertise required to translate these descriptions into the proper syntax, and even experts can struggle to keep up with new developments. Recent advances in large language models (LLMs) offer a possible path forward, serving as interfaces between natural language descriptions and valid LAMMPS inputs. Yet, we lack procedures to evaluate, fix, and test LLM-generated LAMMPS scripts. In this work, we develop and test a procedure to evaluate LLM-generated input files for several MD tasks and assess the performance of widely used LLMs. We evaluate the ability of current general-purpose LLMs to translate natural language prompts into valid input scripts, and in doing so provide a first benchmark of their performance on a realistic scientific DSL without domain-specific fine-tuning. The evaluation procedure starts with a normalization step that generates canonical inputs; this is followed by a static LAMMPS parser for syntax validation and early error detection and correction. Once an input file passes the parser, we use a series of simple tests to identify additional errors and evaluate the accuracy of the generated scripts. The evaluation procedure provides a framework for assessing how LLMs can be deployed in simulation-driven research, with an emphasis on reliability, interpretability, and reproducibility. Beyond LAMMPS, our approach points to a more general strategy for integrating LLMs into DSL-based scientific workflows, where bridging human intent and machine-readable inputs remains a critical challenge. In contrast to studies that focus on fine-tuning or domain-specific retraining of LLMs, our work emphasizes the creation of validation tools and evaluation pipelines that can be extended across DSLs and scientific disciplines. 

# Related Work 

Beyond natural language tasks, LLMs have evolved into powerful problem solvers for both language and code generation. Models like GPT-5, 7 Claude, 8 and Gemini 9 demonstrate state-of-the-art per-formance in general-purpose programming, whether they are generating functional code or tackling domain-specific problems. Researchers have validated these capabilities using standardized bench-marks going from simple coding exercises to realistic bug fixes. 10 This success largely stems from their ability to translate human requests into structured code or data formats, showcasing the versatility of LLMs in conventional programming contexts. Building on these foundational strengths, we are now seeing LLMs applied to concrete applica-tions in science. Recent studies demonstrate that these models, when paired with specialized tools 2or agent-based frameworks, can extend beyond simple text generation to actually support real-world experiments and computational simulations. For instance, projects like CoScientist 11 and Chem-Crow 12 have already demonstrated that LLMs can design and execute wet-lab protocols or plan chemical syntheses by integrating external tools. In computational fields, multi-agent frameworks like AtomAgents 13 and DREAMS 14 are proving that LLMs can plan, run, and analyze complex simulations in physics and materials science. Despite these promising applications, a significant limitation remains: most of these systems remain narrow in scope. They rely heavily on tools and templates carefully curated by humans, rather than demonstrating the flexible, from-scratch generation we can get in general programming tasks. 6, 15 In many cases, AI agents are tasked with preparing scripts for scientific software; how-ever, they operate with expert-prepared packages and predefined templates. 16 As a result, the task becomes less of one of autonomous generation and more one of structured assembly using curated components. This limitation is particularly evident when working with DSLs common in scientific software, which demand a much higher degree of transparency and correctness on their syntax than typical programming tasks. 6, 17 Ultimately, this reliance on template-driven frameworks high-lights the core difficulty of translating natural-language intent into the rigid syntax and semantic requirements of scientific DSLs. Scientific DSLs present a distinct challenge for LLMs. These specialized input languages, used in MD, quantum chemistry, or multi-scale simulators, are highly sensitive to syntax and remain underrepresented in the training of state-of-the-art models. Unlike general-purpose languages such as Python or R, which benefit from compilers, linters, and analysis tools that catch errors early, scientific DSLs typically lack such infrastructure. 6 As a result, even minor mistakes in AI-generated scripts can crash simulations or produce results that look plausible but are scientifically invalid. Jacobs and Pollice 18 demonstrated this in their evaluation of LLMs on ORCA quantum chemistry input files, showing that while models can generate plausible inputs, they often failed to follow domain-specific conventions and syntax, especially without fine-tuning. Similarly, the FEABench project 19 found that LLMs struggled to generate reliable input scripts for finite-element analysis software such as COMSOL. Together, these findings underscore how far current systems are from producing trustworthy input scripts for scientific software. Among scientific DSLs, LAMMPS’s input language provides a particularly instructive case. Its complexity illustrates the broader usability challenges facing scientific DSLs. Recent efforts to im-prove accessibility highlight complementary strategies. The LAMMPS-GUI project 20 has lowered the barrier to entry by providing a graphical interface with syntax highlighting, auto-completion, and inline help for configuring and running simulations. AI-based approaches extend this direction: MDAgent, 21 a fine-tuned LLM assistant, can generate and review scripts for thermodynamic prop-erty calculations, suggesting that domain-specific training improves output quality. At the same time, its reliance on researchers to manually inspect inputs, and on the AI model to evaluate its own code, emphasizes the need for independent parsers and linters that can validate DSLs automatically. Pushing automation further, DynaMate 16 provides a modular multi-agent framework that manages the full molecular dynamics workflow from setup to analysis. Instead of writing inputs directly, it coordinates tools that use a Python package to convert high-level arguments into LAMMPS scripts, but this approach remains limited by template-based scaffolding. Together, these projects demon-strate progress in easing LAMMPS usability through graphical interfaces and AI-driven frameworks, while also showing that systematic evaluation and generalization across scientific domains remain open challenges. 3Methods 

To evaluate LLM-generated input files for LAMMPS, we developed and implemented a multi-stage procedure and evaluated it across tasks of varying complexity, as shown in Figure 1. The first stage involves script generation through each model’s public API using default API parameters unless specified otherwise. All models received identical prompts and system instructions to ensure comparability. The system prompt, provided in the Supporting Information, was constructed using established prompt-design strategies such as explicit role specification, structured task instructions, contextual grounding, and chain-of-thought prompting. 22, 23 We note that this setup reflects guided rather than raw zero-shot performance, as would be expected in a realistic use case. These generated input files are evaluated across three progressively complex prompts adapted from prior work. 24 Prompt 1 represents a simple Aluminum single crystal equilibration at room temperature and pressure. Prompt 2 captures continuous heating of a bulk Ni single crystal from 300K to 2500K utilizing the isothermal-isobaric ensemble. Prompt 3, the most complex of the three, outlines a Niobium spall simulation with a projectile-gap-target impact setup. Together, these prompts span a modest range of complexity within the LAMMPS simulation space while focusing on simple single-metal systems rather than sampling the most complex multi-component, reactive, or molecular systems. These prompts were selected to span a representative range of system complexities and simulation objectives. The full text of the three evaluation prompts is shown in the Supporting Information. Post-processing of LAMMPS outputs is commonly performed using custom analysis workflows, so we excluded analysis-specific commands from evaluation to focus on simulation setup and execution logic. For each prompt–model combination, ten independent generations were produced (10-shot sampling) to capture stochastic variability in model outputs. After normalization, the input scripts are processed by lammps-ast , our custom parser for the LAMMPS input language built using the Lark Python package. 25 The parser transforms the script and its LAMMPS commands, keywords, and argument lists into a typed abstract syntax tree (AST) representation. This process mirrors the compilation process for general-purpose languages, where the syntax tree serves as an intermediate representation for static analysis and semantic validation. By operating on a hierarchical tree representation rather than on linear text, the parser enables programmatic operations such as syntax inspection (for example, argument validation) and semantic consistency checking (for example, cross-referencing regions and variables). This approach allows malformed commands and incorrect arguments to be detected prior to MD simulation execution, so that syntactic inconsistencies can be identified and reported prior to simulation submission. A portion of an AST for these scripts can be seen in Figure 2. This details the workflow from documentation to tree. Importantly, the current version incorporates a subset of the LAMMPS commands, but as a Github repository, it can evolve with user needs or documentation changes. After parsing, all ‘run’ statements in the scripts are adjusted to execute only 10 steps. This reduction limits the computational cost while still capturing most readily observable execution errors. For each script, the outcome of the LAMMPS run is recorded, and, in case of failure, the final line of the log file is captured as the error signal. This execution step also captures errors related to command ordering, which is not handled by the parser. One major source of failure during this execution step arises from inconsistencies during the setup of the interatomic potential used to describe interactions ( pair style parameters). Because these errors occur frequently and can obscure other issues, we introduced an additional step in which all pair style commands are replaced with 

“pair style zero” (PSZ). This substitution separates errors related to syntax and variable handling from those tied to interatomic potential specification. After this modification, a second round of reduced-step execution is performed to continue assessing script executability and isolating sources of error. Finally, for scripts that executed without errors, either in their original form or after the PSZ substitution, we evaluated accuracy using prompt-specific checklists of required simulation con-ditions. For each prompt, we constructed quantitative criteria for the key physical parameters specified in the description by defining target values with an allowed tolerance. These criteria were organized around several fundamental aspects of the simulation setup. System definition included verification of the lattice parameters used to build the systems, boundary conditions, and the di-mensions of the simulation cell. For the thermodynamic settings, we verified the ensemble used and the associated target temperatures and pressures. We also assessed the dynamics parameters, including the choice of timestep and the damping constants that regulate the system’s relaxation behavior. Finally, for the execution parameters, we examined run durations, heating or cooling rates, and the procedures used to initialize particle velocities. Each input file was evaluated against these criteria to determine whether the simulation would achieve the expected state or behavior. The criteria were iteratively refined to ensure coverage of all failure modes. This checklist-based 6assessment provides a systematic and reproducible measure of physical correctness that extends beyond mere syntactic or executability checks. 

# Results 

We evaluated a total of 150 LLM-generated LAMMPS input scripts across all three prompts and five language models. These scripts were evaluated using the multi-stage assessment pipeline consisting of normalization, static parsing, reduced-step execution, and simulation accuracy checks. This pipeline allowed us not only to quantify the overall success rates, but also to identify failures. Across all models and prompts, approximately 74% of the scripts passed the parser, with a smaller subset (32%) executing without error on the first attempt, see Figure 3. Just over a quarter (27.3%) of the scripts satisfied all the physical criteria specified in the prompts, with the majority corresponding to Prompt 1. This sharp drop in performance as the tasks become more complex illustrates the difficulty these LLMs face when generating inputs for scientific DSLs. Figure 3 shows the number of scripts that pass each of the steps in our evaluation process, and a detailed breakdown for every prompt-model combination is provided in Table S1. Although only a relatively small fraction of scripts satisfy all simulation requirements end-to-end, the majority contain few errors and represent a good starting point towards setting up the desired simulation. The results stress the need for a rigorous evaluation procedure for DSLs generated by LLMs.

To compare model behavior across the full LAMMPS code-generation pipeline, for all prompts, we evaluate each model using three aggregate metrics: one-shot accuracy (the fraction of scripts that were fully correct without modification), parser pass rate (the fraction that passed our normal-ization and static parser), and execution success rate (the fraction that completed a full LAMMPS run, including scripts that required only a PSZ correction). Execution success refers to successful completion of a LAMMPS run and does not guarantee the simulation matches the desired out-come described in the prompt. As shown in Table 2, Claude Opus 4 shows the strongest overall performance due to a combination of having the most reliable parser outcomes, one of the best execution success rates, and the second-highest accuracy. Its failures occur primarily in Prompt 2 and Prompt 3, where incorrect pair style selections and boundary-condition errors cause runs to stop after only a few steps. GPT-4.1 and GPT-4o perform well on the simplest prompt but degrade 7sharply on more complex workflows. Their most common errors involve incorrect pair style usage, malformed math expressions, and invalid region or group definitions used by LAMMPS to build atomistic models. Even so, both maintain high parser pass rates and reasonable execution success, with many runs completing after only a minor PSZ fix. In contrast, GPT-o3 and GPT-5 show reduced robustness across the pipeline: GPT-5 produces most normalizer failures due to malformed variable expressions, while GPT-o3 exhibits the weakest parser and execution rates overall. Com-mon issues for both models include hallucinated command arguments, invalid variable expressions, and structural inconsistencies that prevent scripts from parsing or running. Notably, for Prompt 3, GPT-o3 produced one nearly correct script with a single remaining error, whereas GPT-5 was the only model to generate a fully accurate script for the most complex prompt, and ended up having the highest one-shot accuracy rate of any of the models.

# Discussion 

We introduce a procedure to evaluate LLM-generated input files for LAMMPS. The multi-step procedure allows the identification of errors. Our results show that current LLMs can often generate 9syntactically valid LAMMPS input scripts and capture key elements of standard MD workflows. However, performance degrades sharply as the simulation complexity increases and tasks require a larger number of commands, more intricate mathematical specifications, and increasingly complex geometries. These findings indicate that present-day LLMs, when used in isolation, are not yet able to reliably interpret and implement the multi-constraint reasoning required for end-to-end scientific simulations. Many failures appear as incorrect or incomplete commands, and our analysis shows that such issues rarely happen in isolation. Errors in pair style definitions, for example, frequently coincide with additional inaccuracies in command argument specifications. In more complex prompts, models also struggle with geometric reasoning, including the consistent definition of regions, groups, and spatial relationships. These patterns suggest that the observed limitations arise from a combination of the complexity of the LAMMPS DSL, the limited availability of LAMMPS-specific training data, and the difficulty models face in reasoning over symbolic commands and physical constraints within a coherent workflow. Aggregate success rates alone do not fully explain why LLMs fail on complex scientific work-flows. To gain insight into the underlying causes of these failures, we describe a set of representative case studies that expose several common failure mechanisms, progressing from errors in command selection to breakdowns in numerical grounding, semantic validity, and multi-constraint physical rea-soning. The most frequently observed failure mode in our study was incorrect pair style definitions, affecting nearly a third of the 150 generated input scripts. This pattern arises from a combination of factors. First, embedded atom model (EAM) potentials are implemented in LAMMPS through mul-tiple, closely related parameterizations, including ‘eam’, ‘eam/alloy’, ‘eam/fs’, which differ subtly in usage despite sharing similar underlying mathematical formulations. Second, LAMMPS addi-tionally supports EAM potential definitions through the OpenKIM 27 database, introducing more variability in how it can be specified. Finally, the natural language user prompt did not explicitly constrain which EAM variant to use, as the selection of the appropriate potential (e.g., ‘eam/alloy’ for metallic systems) is typically treated as implicit domain knowledge rather than a user-specified instruction. We stress that verification and validation 28 remain critical steps in all physics-based simulations and note that once a working script is generated, LLMs can accelerate their modification to perform validation tests, such as energy and momenta conservation. Another common failure mode involved the use of placeholder or default values in place of task-specific parameters. In several generated scripts, models reverted to generic values when required to infer material-dependent or unit-sensitive quantities from the prompt. One representative ex-ample is the specification of lattice parameters, which requires knowledge of the element(s) being simulated. When prompted to construct a crystal, models were expected to supply an appropriate lattice constant; however, in many cases it was set to 1 ˚ A, a generic value rather than a physi-cally meaningful choice. Similar behavior was observed for other parameters, including simulation timestep and barostat or thermostat damping constants, which were occasionally set to default val-ues or deviated from reasonable physical scales by orders of magnitude. As an example, in Prompt 3, projectile velocities were frequently specified as 2000 without an appropriate unit conversion, suggesting that some models implicitly assumed SI units (m/s) rather than the ˚ A/ps units required under ‘units metal’. This failure mode highlights a broader limitation: when required to translate physical quantities across units, scales, and material-specific properties, models often fail to perform the necessary contextual reasoning. A third recurring error was the generation of command/argument pairings that appear syntac-tically plausible, but are invalid in the LAMMPS DSL. In a subset of Prompt 3 scripts, models hallucinated a non-existent ‘velocity’ style where they used ’velocity groupID add’ to assign the 10 impact velocity of the projectile, rather than the correct syntax ‘velocity groupID set vx vy vz sum yes’. This command variant is not supported by LAMMPS and does not appear in the official doc-umentation. Notably, we found only a single reference to this construction in an informal discussion forum, 29 suggesting that while the syntax is not valid, it reflects a natural way users might attempt to express incremental velocity assignment. This behavior suggests that models may extrapolate command semantics based on perceived usage patterns rather than formal language definitions. While such hallucinations undermine semantic validity, they also point to potential ambiguity be-tween how users conceptually describe operations and how they are expressed in the LAMMPS DSL. The most severe breakdowns were observed in Prompt 3, which requires coordinating multi-ple interdependent assumptions to construct a physically consistent simulation. Unlike the earlier prompts, several critical choices are not explicitly specified and must be inferred from context. One such example is the treatment of boundary conditions along the shock direction. While the prompt requests free boundaries, this must be translated in LAMMPS into either fixed or shrink-wrapped non-periodic boundaries. Although both choices are formally valid, a fixed boundary would re-quire extending the simulation box to accommodate high-velocity atoms, whereas a shrink-wrapped boundary does not. Selecting the appropriate option therefore requires anticipating downstream physical consequences rather than just applying a local syntax rule. Similar multi-step reasoning is required for geometric setup and unit translation. Distances in the prompt are expressed using a mixture of lattice units and nanometers, while the ‘units metal’ command interprets all lengths in angstroms and requires the lattice constant to be inferred from the material. These quantities must then be consistently translated into geometries and region definitions with correct command ordering. Across models, this combination of unit conversion, geometric reasoning, and command sequencing proved difficult to resolve reliably. While isolated components of the setup were often handled correctly, failures frequently emerged from inconsistencies across these coupled steps. The limited success observed for the more complex prompts indicates that model accuracy degrades rapidly as simulation workflows demand tighter coupling between syntax, numerical spec-ification, geometry, and physical intent. While these tasks already expose substantial limitations, they remain simpler than many simulations routinely performed by MD practitioners, suggesting these challenges will be amplified in realistic research settings. At lower levels of complexity, many failures are localized and structurally identifiable, such as mismatched command usage or incom-plete argument specification. In these cases, domain-aware parsing and static validation, such as the parser introduced in this work, can serve as effective tooling to detect errors, enforce structural constraints, and support iterative refinement of generated scripts. In contrast, failures in more com-plex workflows frequently arise from inconsistencies spanning multiple interdependent steps, which cannot be resolved through syntax or structural checks alone. These results suggest that language models are best viewed not as autonomous simulation designers, but as assistive components within workflows that combine generation with validation, feedback, and expert oversight. 

# Future Work 

Developing an autonomous workflow capable of translating natural language physics descriptions into reliable MD simulations requires addressing the gap between runnable code and scientifically valid workflows. LLMs frequently generate syntactically correct scripts; however, producing code that fully captures the intended physical and scientific constraints remains a substantial challenge. 24 

In scientific DSLs such as LAMMPS, correctness is not determined solely by syntax but also by the consistent handling of units, geometries, boundary conditions, and material-specific parameters, as 11 well as by the semantically correct ordering of commands. In practice, model performance is highly sensitive to prompt formulation and iterative refinement. While users often improve outcomes through experimentation and prompt adjustments, this process assumes both domain expertise and an understanding of the LLM’s behavior. Achieving robust automation therefore requires shifting from prompt-driven refinement toward structured systems that explicitly detect and correct model assumptions. Unlike common pro-gramming languages such as Python or HTML, scientific DSLs lack mature compiler and linting infrastructures that provide early error detection. In these general-purpose languages, syntactic and structural issues are typically identified before or during inexpensive execution, allowing rapid itera-tive refinement. In contrast, the limited availability of static validation and the high computational cost of execution in MD workflows drastically limits naive LLM-driven iteration. Consequently, LLM-generated LAMMPS scripts face four structural barriers to reliable automation: the absence of robust pre-execution validation; runtime error signals that are often insufficiently informative for automated correction; the lack of automated mechanisms for assessing scientific correctness beyond syntax and execution; and the substantial computational expense of meaningful simulations, which limits large-scale iterative refinement. To address these structural limitations, we have developed a prototype workflow that integrates script generation, validation, and execution into a unified research pipeline, as illustrated in Figure 5.

The static parser is integrated as a pre-execution validation layer, ensuring that generated scripts satisfy syntax and command-level correctness before simulation submission. Although the workflow does not fully automate scientific validation or eliminate the need for expert oversight, it establishes a framework for integrating generation, validation, and execution within a unified research pipeline. Formal mechanisms for assessing physical accuracy remain an open challenge across domains, underscoring the need for architectures in which model assumptions are either explicitly constrained or systematically verified through structured validation 12 tools. Future extensions of this framework may incorporate documentation-grounded retrieval mech-anisms to reduce hallucinated command usage and improve alignment with LAMMPS syntax and potential specifications. More broadly, integrating the parser as an actionable tool within an agen-tic generation-validation loop could enable iterative script refinement based on structured syntax checks and runtime diagnostics. Such architectures would move beyond prompt refinement toward self-correcting scientific code generation systems. In parallel, recent work has also explored in-corporating structural representations, such as ASTs, directly into model pre-training to improve syntactic reliability in code generation tasks. 32 Combining structure-aware pretraining with down-stream validation and execution-aware feedback represents a promising direction for building more robust scientific DSL pipelines. 

# Conclusions 

In this work, we introduce a procedure to evaluate LAMMPS input scripts generated by language models from natural language descriptions. While the models tested frequently produce syntac-tically plausible and partially correct scripts, our results show that performance and reliability degrade sharply as simulation workflows increase in structural and physical complexity. Common failure modes include incorrect pair style selection, inconsistent unit handling, malformed command arguments, and breakdowns in multi-step physical reasoning. These findings demonstrate that, in their base configurations, LLMs are not yet capable of serving as fully autonomous scientific sim-ulation designers. While newer models will likely improve on these results, systematic evaluation remains essential, as more plausible outputs make undetected errors increasingly difficult to catch. However, they consistently generate useful starting points and often capture the high-level structure of MD workflows. When paired with structured validation tools, such as the static AST-based parser introduced here, many subtle errors can be identified prior to costly simulation execution. The structure-aware parser developed in this work enables early syntax validation and programmatic analysis of LAMMPS scripts, allowing structural inconsistencies and invalid command constructions to be detected before runtime. By explicitly separating structural correctness from physical validity, this framework establishes a foundation for more robust AI-assisted simulation pipelines and offers a practical pathway toward integrating LLMs into domain-specific computational ecosystems while preserving reliability and reproducibility.
